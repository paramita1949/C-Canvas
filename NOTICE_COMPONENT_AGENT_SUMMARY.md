# 通知组件构建与修复总结（Agent 参考）

## 1. 目的
记录通知组件从构建到修复的关键过程，沉淀后续开发可复用的方法，重点避免再次误判左右滚动问题。

适用场景：
- 文本通知滚动（`LeftToRight` / `RightToLeft` / `PingPong`）
- 多画布比例（16:9、4:3、自定义）
- 任何“位移 + 边界 + 回卷”动画能力

## 2. 最终行为定义
单向滚动统一为：
- `R->L`：文字**完全离开左侧画布**后，从右侧重新开始。
- `L->R`：文字**完全离开右侧画布**后，从左侧重新开始。

即：不是“触边回卷”，而是“全部消失后回卷”。

## 3. 构建结构
- 位移/回卷核心：`Services/TextEditor/Components/Notice/NoticeRuntimeService.cs`
- UI 偏移接入：`UI/MainWindow.TextEditor.Helpers.cs`
- 行为单测：`Canvas.TextEditor.Tests/Components/NoticeRuntimeServiceTests.cs`

核心参数：
- `viewportWidth`：可视宽度（必须来自实际画布）
- `contentWidth`：内容宽度
- `laneStartX` / `laneEndX`：轨道左右内边距
- `initialContentX`：初始位置

## 4. 关键修复复盘
### 阶段 A：修复坐标系不一致（真正根因）
现象：`R->L` 看起来正常，`L->R` 不正常（像“消失很久后才回弹”）。

根因不是方向公式，而是**编辑框/项目宽度与实际画布宽度不一致**：
- 项目宽度常见为 `1920`
- 实际画布在 16:9/4:3 下可能是 `1600` 或 `960`

结果：`L->R` 的右边界判定被推迟到画布外，导致视觉异常。

修复：
- `viewportWidth` 优先使用 `EditorCanvas.ActualWidth`，回退才用项目宽度。

### 阶段 B：行为语义调整
需求最终明确为“全部消失后再开始”。

落地变化：
- `L->R` 从“右边缘触边回卷”改为“整段离场回卷”
- 关键公式在 `NoticeRuntimeService` 调整为：`L->R` 行程按 `laneRight - startX`

## 5. 最重要经验
1. 先验尺寸一致性，再看方向算法。  
2. 一个方向“看起来正常”不能证明系统正确。  
3. 边界逻辑必须绑定需求语义：触边回卷 vs 离场回卷。  

## 6. 测试策略（后续功能强制复用）
单测必须覆盖：
- `L->R` 全离场回卷
- `R->L` 全离场回卷
- 带 `contentStartX` 的内边距场景
- 真实视口（如 `1920/1600/960`）
- `PingPong` 无回归

UI 验收必须覆盖：
1. 16:9 与 4:3
2. 短文本与长文本
3. `L->R` 与 `R->L`
4. 切分辨率后再次播放

## 7. 新功能参考模板
开发“运动+边界”新功能时按以下顺序：
1. 先定义语义：触边回卷 or 全离场回卷
2. 统一坐标系：实际渲染容器尺寸
3. 先写失败测试，再写实现（TDD）
4. 做双方向对称验证
5. 做多比例回归验证

检查清单：
- [ ] 视口尺寸来自实际容器（非仅项目配置）
- [ ] 内容测宽与视觉一致
- [ ] 回卷阈值和需求语义一致
- [ ] 覆盖“看似正常但实际错误”的伪通过场景
