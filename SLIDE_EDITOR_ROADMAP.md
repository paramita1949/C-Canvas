# 咏慕投影 - 幻灯片文本编辑功能改进规划

## 1. 当前功能评估

### 1.1 已实现功能清单
目前系统已经建立了一个基于 WPF + SkiaSharp 的高性能幻灯片编辑框架，核心功能包括：
- **基础编辑**: 元素的增删、自由拖拽、8方向缩放、编辑模式切换。
- **文本样式**: 完善的字体、字号、颜色、粗斜体、下划线、对齐、行间距及字间距设置。
- **高级视觉**: 支持边框（圆角/透明度）、背景（圆角/透明度）、阴影（模糊/偏移/透明度）。
- **富文本**: 同一个文本框内支持多种样式片段，基于 `RichTextSpan` 管理。
- **布局管理**: 灵活的 ZIndex 层级控制、多重画面分割模式、对称联动功能。
- **投影系统**: 实时高性能投影渲染，支持动画过渡和视频背景。

### 1.2 功能完成度分析
当前版本已满足“可用”标准，但在“好用”和“高效”上仍有较大提升空间。现有的操作逻辑主要集中在单元素的操作上，缺乏批量处理和辅助对齐的手段。

### 1.3 用户反馈痛点
- **误操作难恢复**: 缺少撤销/重做，一旦删错或移动错位无法快速恢复。
- **布局对齐耗时**: 纯靠肉眼观察和手动拖拽，难以实现多个元素间的精确对齐。
- **操作繁琐**: 缺乏快捷键支持，如全选、复制粘贴等均需通过右键菜单或多次点击。
- **层级管理困难**: 当页面元素较多时，难以准确选中被压在底层的元素。

## 2. 缺失功能分析

### 2.1 必要功能（高优先级）
这些功能是现代化编辑器的“标配”，能显著降低用户的挫败感。
- **撤销/重做 (Undo/Redo)**: 解决误操作问题。
- **快捷键支持 (Hotkeys)**: Ctrl+C/V/Z/Y/A/S 等。
- **对齐工具**: 一键实现左对齐、居中、等间距等。
- **删除/移动快捷键**: 使用 Del 键删除，方向键微调位置。

### 2.2 实用功能（中优先级）
提升编辑效率的关键功能。
- **多选功能**: 支持 Ctrl 点击多选和鼠标框选。
- **图层面板**: 可视化管理元素，支持锁定和隐藏。
- **吸附功能 (Snapping)**: 拖拽时自动吸附到边框或中心线。
- **格式刷**: 快速复制一个元素的完整样式到另一个元素。

### 2.3 增强功能（低优先级）
锦上添花，进一步提升用户体验。
- **画布缩放**: 方便编辑细微处。
- **标尺与参考线**: 提供精确的视觉参考。
- **自动保存**: 防止程序崩溃或停电导致的数据丢失。

## 3. 详细实现方案

### 3.1 撤销/重做 (Undo/Redo)
- **优先级**: 高
- **用户价值**: 消除误操作焦虑，提升编辑大胆度。
- **实现方案**:
  1. **数据模型**: 维护两个 `Stack<SlideSnapshot>`。
  2. **快照机制**: 每当执行关键操作（添加、删除、移动结束、样式修改结束）时，将当前 Slide 的序列化数据压入撤销栈。
  3. **交互**: 监听 Ctrl+Z 和 Ctrl+Y。
- **开发工作量**: 2人天

### 3.2 对齐与分布工具
- **优先级**: 高
- **用户价值**: 快速实现整齐美观的排版。
- **实现方案**:
  1. **UI设计**: 在侧边栏或顶部工具栏添加对齐图标按钮组。
  2. **逻辑实现**: 
     - **左对齐**: 取所有选中元素中最小的 X 值，将其他元素的 X 设为该值。
     - **垂直居中分布**: 计算总高度，平分间距。
  3. **数据更新**: 批量更新 `TextElement` 的 X, Y 坐标并同步到数据库。
- **开发工作量**: 1.5人天

### 3.3 多选功能与批量操作
- **优先级**: 中
- **用户价值**: 一次修改多个元素的字体、颜色等。
- **实现方案**:
  1. **状态管理**: 将 `_selectedTextBox` 修改为 `List<DraggableTextBox> _selectedTextBoxes`。
  2. **框选逻辑**: 在 Canvas 上监听鼠标拖拽，生成一个临时蓝色半透明矩形，计算与其相交的元素。
  3. **批量更新**: 修改样式时，遍历 `_selectedTextBoxes` 调用 `ApplyStyle`。
- **开发工作量**: 3人天

### 3.4 图层管理面板
- **优先级**: 中
- **用户价值**: 清晰查看元素层级，解决底层元素选中难的问题。
- **实现方案**:
  1. **UI设计**: 右侧新增“图层”选项卡，使用 `ListBox` 展示。
  2. **交互**: 支持点击选中、拖拽排序（对应 ZIndex）、点击眼睛图标（Visibility）、点击锁图标（IsEnabled）。
- **开发工作量**: 2人天

## 4. 技术可行性分析

### 4.1 现有技术栈支持
- **WPF**: 本身具备强大的事件处理和数据绑定能力，非常适合实现多选框、图层面板等。
- **SkiaSharp**: 仅负责渲染预览和投影，不影响编辑逻辑。当前的 `DraggableTextBox` 已经是包装好的控件，扩展性强。
- **SQLite**: 支持事务，适合实现批量更新和自动保存逻辑。

### 4.2 性能影响评估
- **撤销快照**: 内存占用极小（一个幻灯片的数据量通常在几十KB以内），限制栈深度为50步即可。
- **吸附功能**: 在 `OnMouseMove` 中计算，只需遍历当前页面的少量元素（通常 < 20个），计算开销可忽略。
- **多选渲染**: 多个控制点的显示对 GPU 几乎无压力。

## 6. 实施路线图

### 6.1 第一阶段：基础稳固（3-4天）
- 实现撤销/重做管理器。
- 绑定基础快捷键（Ctrl+Z, Ctrl+Y, Ctrl+C, Ctrl+V, Delete）。
- 实现单个元素的键盘微调（方向键）。

### 6.2 第二阶段：排版增效（4-5天）
- 开发多选逻辑（Ctrl多选 + 框选）。
- 实现对齐与分布工具栏。
- 实现元素拖拽时的智能参考线吸附。

### 6.3 第三阶段：深度管理（3-4天）
- 开发可视化图层面板。
- 实现样式格式刷。
- 优化右键菜单，加入层级置顶/置底快捷操作。

---

## 7. 风险与注意事项
1. **数据一致性**: 批量操作和撤销时，确保 UI 状态、内存 Data 状态和数据库三者完全同步。
2. **富文本复杂度**: 批量修改样式时，需注意对已有 `RichTextSpan` 的合并或覆盖逻辑，避免产生冗余数据。
3. **操作冲突**: 在编辑模式（打字中）时，需暂时禁用全局快捷键，避免输入 Z 时触发撤销。
