# 先诊断后修改（禁止盲改）规则

适用范围：`UI/`、`Managers/`、`Database/` 及所有线上行为相关代码。

## 核心原则
- 先拿证据，再改代码。
- 一次只验证一个假设。
- 没有可复现步骤和观测点，不提交“修复”。

## 强制流程
1. 定义问题
- 写清楚：现象、触发条件、期望行为、实际行为。

2. 建立最小假设
- 每次只提出 1 个根因假设，不允许“打包猜测”。

3. 添加最小观测
- 优先使用临时 `#if DEBUG` 日志或断点观测。
- 日志必须包含：关键状态值、分支路径、触发时机。

4. 小步改动
- 每次提交只对应 1 个假设，避免混改。
- 不改无关模块，不顺手重构。

5. 验证与回归
- 至少验证：问题复现场景 + 一个相邻场景。
- 给出“修复证据”：前后日志、关键状态对比、构建结果。

6. 清理临时诊断
- 任务结束后删除临时日志/临时开关。
- 保留长期日志需说明目的与开销。

## 禁止项
- 禁止在无法复现时直接改业务逻辑碰运气。
- 禁止一次提交同时包含“猜测修复 + 样式调整 + 重构”。
- 禁止没有证据就下“已修复”结论。

## 提交说明模板（必须）
```
问题现象：
根因假设：
观测证据：
改动点：
验证结果：
回归影响：
```
